ストレージをアップグレードする前に、Elementストレージのヘルスチェックを実行します

2020年9月29日寄稿者このページのPDFをダウンロード

Elementストレージをアップグレードする前にヘルスチェックを実行して、クラスター内のすべてのストレージノードが次のElementストレージアップグレードの準備ができていることを確認する必要があります。

必要なもの

  • 最新の管理サービスバンドル（2.10.27以降）に更新しました。

    Elementソフトウェアをアップグレードする前に、最新の管理サービスバンドルにアップグレードする必要があります。

  • 管理ノード11.3以降を実行しています。

  • クラスタバージョンは、NetAppElementソフトウェア11.3以降を実行しています。

ヘルスチェックオプション

ネットアップハイブリッドクラウドコントロール（HCC）UI、HCC API、またはHealthToolsスイートを使用してヘルスチェックを実行できます。

  • NetApp Hybrid Cloud Controlを使用して、ストレージをアップグレードする前にElementストレージのヘルスチェックを実行します（推奨される方法）

  • ストレージをアップグレードする前に、APIを使用して要素ストレージのヘルスチェックを実行します

  • ストレージをアップグレードする前に、HealthToolsを使用して要素ストレージのヘルスチェックを実行します

また、サービスによって実行されるストレージヘルスチェックの詳細を確認することもできます。

  • サービスによって行われるストレージヘルスチェック

ストレージをアップグレードする前に、NetApp Hybrid CloudControlを使用してElementストレージのヘルスチェックを実行します

NetApp Hybrid Cloud Control（HCC）を使用すると、ストレージクラスターをアップグレードする準備ができていることを確認できます。

ステップ

  1. Webブラウザーを開き、管理ノードのIPアドレスを参照します。

     https：// <ManagementNodeIP>

  2. ストレージクラスター管理者の資格情報を入力して、NetApp Hybrid CloudControlにログインします。

  3. インターフェイスの右上にある[アップグレード]をクリックします。

  4. [アップグレード]ページで、[ストレージ]タブを選択します。

  5. ヘルスチェックをクリックしますアップグレードの準備ができているかどうかを確認するクラスターの場合。

  6. [ストレージヘルスチェック]ページで、[ヘルスチェックの実行]をクリックします。

  7. 問題がある場合は、次のようにします。

    1. 各問題にリストされている特定のKB記事に移動するか、指定された対処法を実行してください。

    2. KBが指定されている場合は、関連するKBの記事に記載されているプロセスを完了してください。

    3. クラスタの問題を解決したら、[ヘルスチェックの再実行]をクリックします。

ヘルスチェックがエラーなしで完了すると、ストレージクラスターをアップグレードする準備が整います。続行するには、ストレージノードのアップグレード手順を参照してください。

ストレージをアップグレードする前に、APIを使用して要素ストレージのヘルスチェックを実行します

REST APIを使用して、ストレージクラスターをアップグレードする準備ができていることを確認できます。ヘルスチェックは、保留中のノード、ディスクスペースの問題、クラスターの障害など、アップグレードに障害がないことを確認します。

ステップ

  1. ストレージクラスターIDを見つけます。

    1. 管理ノードで管理ノードRESTAPIUIを開きます。

       https：// [管理ノードIP] / Inventory / 1 /

    2. [承認]をクリックして、以下を完了します。

      1. クラスターのユーザー名とパスワードを入力します。

      2. 値がまだ入力されていない場合は、クライアントIDをmnode-clientとして入力します。

      3. [承認]をクリックしてセッションを開始します。

      4. 承認ウィンドウを閉じます。

    3. REST API UIから、[ GET / installations ]をクリックします。

    4. [試してみる]をクリックします。

    5. [実行]をクリックします。

    6. 応答から、インストール資産ID（ "id" ）をコピーします。

    7. REST API UIから、[ GET / settings / {id} ]をクリックします。

    8. [試してみる]をクリックします。

    9. インストール資産IDをidフィールドに貼り付けます。

    10. [実行]をクリックします。

    11. 応答から、アップグレードの準備ができているかどうかを確認する予定のクラスターのストレージクラスターID（ "id" ）をコピーして保存します。

  2. ストレージクラスターでヘルスチェックを実行します。

    1. 管理ノードでストレージRESTAPIUIを開きます。

       https：// [管理ノードIP] / storage / 1 /

    2. [承認]をクリックして、以下を完了します。

      1. クラスターのユーザー名とパスワードを入力します。

      2. 値がまだ入力されていない場合は、クライアントIDをmnode-clientとして入力します。

      3. [承認]をクリックしてセッションを開始します。

    3. POST / health-checksをクリックします。

    4. [試してみる]をクリックします。

    5. パラメータフィールドにストレージクラスタIDを入力します。

    6. [実行]をクリックして、指定したストレージクラスターでヘルスチェックを実行します。

      応答は、状態をinitializingとして示す必要がありinitializing 。

       {{
  "_links"：{
    "コレクション"： "https://10.117.149.231/storage/1/health-checks"、
    "ログ"： "https://10.117.149.231/storage/1/health-checks/358f073f-896e-4751-ab7b-ccbb5f61f9fc/log"、
    「自己」：「https://10.117.149.231/storage/1/health-checks/358f073f-896e-4751-ab7b-ccbb5f61f9fc」
  }、
  "config"：{}、
  "dateCompleted"：null、
  "dateCreated"： "2020-02-21T22：11：15.476937 + 00：00"、
  "healthCheckId"： "358f073f-896e-4751-ab7b-ccbb5f61f9fc"、
  「状態」：「初期化中」、
  「ステータス」：null、
  "storageId"： "c6d124b2-396a-4417-8a47-df10d647f4ab"、
  "taskId"： "73f4df64-bda5-42c1-9074-b4e7843dbb77"
}

    7. 応答の一部であるhealthCheckIDをコピーします。

  3. ヘルスチェックの結果を確認します。

    1. GET / health-checks / {healthCheckId}をクリックします。

    2. [試してみる]をクリックします。

    3. パラメータフィールドにヘルスチェックIDを入力します。

    4. [実行]をクリックします。

    5. 応答本文の一番下までスクロールします。

  4. 戻りmessageがクラスターの正常性に関する問題があったことを示している場合は、以下を実行します。

    1. 各問題にリストされている特定のKB記事に移動するか、指定された対処法を実行してください。

    2. KBが指定されている場合は、関連するKBの記事に記載されているプロセスを完了してください。

    3. クラスタの問題を解決したら、 GET / health-checks / {healthCheckId}を再度実行します。

すべてのヘルスチェックが成功した場合、戻り値は次の例のようになります。

 "メッセージ"： "すべてのチェックが正常に完了しました。"、
「パーセント」：100、
「タイムスタンプ」：「2020-03-06T00：03：16.321621Z」

ストレージをアップグレードする前に、HealthToolsを使用して要素ストレージのヘルスチェックを実行します

sfupgradecheckコマンドを使用して、ストレージクラスターをアップグレードする準備ができていることを確認できます。このコマンドは、保留中のノード、ディスクスペース、クラスター障害などの情報を確認します。

管理ノードが暗いサイトにある場合、アップグレードの準備チェックを正常に実行するには、 HealthToolsのアップグレード中にダウンロードしたmetadata.jsonファイルが必要です。

このタスクについて

この手順では、次のいずれかの結果をもたらすアップグレードチェックに対処する方法について説明します。

  • sfupgradecheckコマンドの実行は正常に実行されます。クラスターはアップグレードの準備ができています。

  • sfupgradecheckツール内のチェックは失敗し、エラーメッセージが表示されます。クラスタはアップグレードの準備ができておらず、追加の手順が必要です。

  • アップグレードチェックが失敗し、HealthToolsが古くなっているというエラーメッセージが表示されます。

  • 管理ノードが暗いサイトにあるため、アップグレードチェックは失敗します。

ステップ

  1. sfupgradecheckコマンドを実行します。

     sfupgradecheck -u <cluster-user-name> MVIP
    特殊文字を含むパスワードの場合、各特殊文字の前に円記号（ \ ）を追加します。たとえば、 mypass!@1はmypass\!\@として入力する必要があります。

    エラーが表示されず、アップグレードの準備ができているサンプル出力を含むサンプル入力コマンド：

     sfupgradecheck -u admin 10.117.78.244

HealthToolsは古くなっています

HealthToolsが最新バージョンではないことを示すエラーメッセージが表示された場合は、次の手順に従ってください。

  1. エラーメッセージを確認し、アップグレードチェックが失敗することに注意してください。

    応答例：

     sfupgradecheckが失敗しました：HealthToolsが古くなっています：
インストールバージョン：2018.02.01.200
最新バージョン：2020.03.01.09。
HealthToolsの最新バージョンは、https：//mysupport.netapp.com/NOW/cgi-bin/software/からダウンロードできます。
または、-nオプションを指定して再実行します

  2. 応答に記載されている指示に従ってください。

管理ノードが暗いサイトにあります

  1. メッセージを確認し、アップグレードチェックが失敗することに注意してください。

    応答例：

     sfupgradecheckが失敗しました：利用可能な最新バージョンのhealthtoolsを確認できません。

  2. 管理ノードではないコンピューターのNetAppサポートサイトからJSONファイルをダウンロードし、名前をmetadata.json変更します。

  3. 次のコマンドを実行します。

     sfupgradecheck -l --metadata = <path-to-metadata-json>

  4. 詳細については、ダークサイトの追加のHealthToolsアップグレード情報を参照してください。

  5. 次のコマンドを実行して、HealthToolsスイートが最新であることを確認します。

     sfupgradecheck -u <クラスターユーザー名> -p <クラスターパスワード> MVIP

サービスによって行われるストレージヘルスチェック

ストレージヘルスチェックは、クラスターごとに次のチェックを行います。

名前を確認してください ノード/クラスター 説明

check_async_results

集まる

データベース内の非同期結果の数がしきい値を下回っていることを確認します。

check_cluster_faults

集まる

（要素ソースで定義されているように）クラスター障害をブロックするアップグレードがないことを確認します。

check_upload_speed

ノード

ストレージノードと管理ノード間のアップロード速度を測定します。

connection_speed_check

ノード

ノードがアップグレードパッケージを提供する管理ノードに接続していることを確認し、接続速度を推定します。

check_cores

ノード

ノード上のカーネルクラッシュダンプとコアファイルをチェックします。最近の期間（しきい値7日）にクラッシュが発生した場合、チェックは失敗します。

check_root_disk_space

ノード

ルートファイルシステムに、アップグレードを実行するのに十分な空き領域があることを確認します。

check_var_log_disk_space

ノード

/var/log空き領域が一定の割合の空きしきい値を満たしていることを確認します。そうでない場合、チェックはローテーションし、しきい値を下回るために古いログをパージします。十分な空き領域の作成に失敗した場合、チェックは失敗します。

check_pending_nodes

集まる

クラスタに保留中のノードがないことを確認します。
